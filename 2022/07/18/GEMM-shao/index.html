GEMM (General Matrix Multiplication, 通用矩阵乘法)

为了简便，一般取$\alpha = 1$，$\beta = 0$：

其中，矩阵$\mathbf{A}$为$M \times K$，矩阵$\mathbf{B}$为$K \times N$，因而矩阵$\mathbf{C}$为$M \times N$。

For CPU computing:

  • M = N = 4096, K = 2048
  • $\alpha = 1$, $\beta = 0$
  • CUDA Version: CUDA 11.1 under 20.12-tf1-py3 image
  • GPU: A100-PCIE-40G on BFS
  • Warm Up: 100 iterations
  • Compute: 100 iterations
  • NCU captures the 50th iteration in Compute’s 100 iterations.

0 Base Line:

Each thread corresponds to one element in the matrix C.
Assume grid size (128, 128), block size (32, 32), since matrix C is 4096 x 4096.

从NCU report来看，当前实现eligible warps较少，大量的warps由于latency原因处于等待，查看发现其中占比最大的为Stall LG Throttle，对应load global access。

问题：对于一个thread来说，计算需要读取A的一行，B的一列，Global mem access非常多，GPU上所有的内存中，global mem的访存延迟最大，延迟由大到小排序为 Global mem = Local mem > Shared mem = L1 cache > Register。同一个elemengt要被不同的线程多次访问，数据reusability不好。
解决方法：使用shared mem。然而block size为(32, 32)，即对于这个Block来说，M = N = 32，K = 2048，则所需的shared mem大小为32 32 2048 * sizeof(Element)
shared mem通常没有那么大，不能装下所需的所有数据，因此需要先对矩阵进行切分。

1 Shared Memory

Each block stores the A tile and B tile that needed into shared mem for reuse. In each iteration of k tile, the process is:

  1. Load the data needed from global mem to shared mem.
  2. Inner product within the block, and accumulate the coresponding C[i][j].

Assume grid size (128, 128), block size (32, 32), tile size (32, 32).

数据传输过程

从global mem到shared mem的数据传输是以sector为单位进行的。
1 sector = 32 bytes

element type = float 对应 4 bytes，因此一个warp每次load 32 * 4 bytes = 4 sectors。

此时最大的latency变为Stall MIO Throttle，对应过多的shared mem access instructions。
解决办法：由 1 个thread计算 1 个element转为计算 (4, 4) elements。目的是为了将每个线程的LDS 32 (32)利用同时访问4个elements的方法合并为LDS 128(32)，从而将shared mem instructions变为原来的1/4。

2 Shared Memory Thread Tile (4, 4)

此时Ktile = 32, Mtile = Ntile = 32 * 4
那么单个warp则计算(128, 128)个 C elements。
对应Atile (128, 32), Btile (32, 128)。

M = N = 4096, K = 2048
Assume grid size (32, 32), block size (32, 32)
a block (32, 32) loads Atile (128, 32), Btile (32, 128), and computes Ctile (128, 128).

其中calcInnerBlockByInnerProduct函数负责计算每个线程所对应的(4, 4)矩阵的结果，返回值也是(4, 4)的矩阵。
其中需要对输出矩阵中的每个element都要计算一次对应A(4,1)与B(1,4)的张量积。而内层的k循环则表征32个thread在Ktile中整体的前进。

添加#pragma unroll后，编译器会将对Atile, Btile的LDS 32，自动优化为LDS 128，从而达到有效减少shared mem access instructions，减少warp的Stall MIO Throttle的作用。

这里插播一条新闻：
经过本人及Perkz Zheng一下午的研究，发现nvcc flag -rdc=true，也就是CUDA的分离编译，会一定程度上阻止编译器进行优化，导致上述编译器将LDS 32优化为LDS 128的现象不会发生。
值得一提的是CMakeLists中set(CUDA_SEPARABLE_COMPILATION ON)可起到与nvcc flag -rdc=true相同的作用。

对于LDS 128指令，shared mem一个bank 32 bit，一共32 banks，正常一个warp执行一次LDS 128指令需要4 wavefronts。(Wavefront: 一个request所需要的cycles数)

这里每个线程每次需要访问4个Atile elements，4个Btile elements，一共沿着Ktile前进了32次，因此为:
32 (4 + 4) LDS 32 = 64 * LDS 128

NCU Report中的Floating Point Operations Roofline部分，其中的横坐标计算访存比Arithmetic Intensity [FLOP/byte]中访存的部分只考虑从Global mem到L2 cache。

Num of Instructions = gridDim.x x gridDim.y x Num_warps_per_block x Num_Ktiles x Num_instruction_per_warp
block总数 x 每个block中warp数 x 每个block对应Ktile数 x 每个warp需要执行的Instruction数

从上图中方框圈出的五个指令的变化来看。

  • FFMA (FP32 Fused Multiply and Add): 乘加操作，对于GEMM来讲，无论怎么优化，其计算量是不发生变化的，因此指令数不变。
  • LDS (Shared Mem Load):
    • 这里两种实现block size均为(32, 32)，所要计算的C矩阵为(4096, 4096)，M = N = 4096，K = 2048
    • 第一种实现每个线程负责计算一个element，Grid size (128, 128)，shared mem对A矩阵为LDS 128(8)，MK tile (32, 32), 而对B矩阵为LDS 32(32)，KN tile (32, 32)
      总指令数 = 128 128 32 (2048 / 32) (8 + 32) = 1,342,177,280
      而对于wavefront，一个LDS 128正常对应4个cycles也就是wavefront是4 (这里满足了某特殊条件，获得了double bandwidth，因而此处LDS 128的wavefront为2)，而LDS 32的wavefront为1
      总wavefront = 128 128 32 (2048 / 32) (8 * 2 + 32) = 1,610,612,736
    • 第二种每个线程计算(4, 4) elements, Grid size (32, 32), shared mem对A矩阵为LDS 128(32), MK tile (128, 32), 而对B矩阵也为LDS 128(32), KN tile (32, 128)，总指令数 = 32 32 32 (2048 / 32) (32 + 32) = 134,217,728 (-90%)
      此处只有对A矩阵的LDS 128满足了double bandwidth的条件
      总wavefront = 32 32 32 (2048 / 32) (32 2 + 32 4) = 402,653,184 (-75%)
  • STS (Store to Shared Mem) & LDG (Global Mem Load): 这两个指令在这里是一对一的关系，由于Mtile与Ntile均 x 4，而Ktile不变，相比于上一个实现，shared mem从global mem中加载的数据量增大了4倍。对于这4部分，原来在Global mem中，A中的每个部分都要被B中的这四部分重复访问四次 (需要从Global mem中重复加载四回)，同理B中的每个部分也要被A中的这四部分重复访问四次 (需要从Global mem中重复加载四回)。因此shared mem预载量增大四倍后，相应的STS与LDG指令减小为原来的1/4。
  • STG (Store to Global Mem): 对应每个线程计算完毕之后写回Global mem的过程，前后不变。

3 STG.32 -> STG.128 (float4)

这里由于每个thread计算对应于C矩阵的一个(4, 4)的小方阵，因此在写回Global mem时，线程间写入位置有一个为4的stride，导致这里对Global mem的uncoalesced access，解决办法是每次显示指定写回一个float4，将STG 32优化为STG 128，可以从NCU Report中看到STG指令数减少了75%。

Global access是以sector (32 bytes)为单位进行的。
正常向global mem写入C矩阵，需要4096 4096 4 = 67,108,864 bytes = 2,097,152 sectors
由于uncoalesced global access，使得同一片区域被重复access四次 (同一个sector被access了四次，每次访问仅用到其32 bytes中的8 bytes)。改用float4后，线程间访存地址连续，因此可见global store sectors下降了75%。

4 Shared Memory Thread Tile (8, 4)

进一步增加每个thread所处理的element个数，保持grid size (32, 32)不变，则block size变为(4096/32/4 = 32, 4096/32/8 = 16)，此时
Mtile = 16 8 = 128
Ntile = 32 4 = 128
Perkz的实现中此处一个block loads (128, 16)的Atile与(16, 128)的Btile，计算(128, 128)的Ctile。因此所需的shared mem size变为原来的一半 (原来Atile与Btile大小均为128 x 32)。

5 Shared Memory Thread Tile (16, 4)

grid size (32, 32)
block size (4096/32/4 = 32, 4096/32/16 = 8)
一个block loads (128, 8)的Atile与(8, 128)的Btile，计算(128, 128)的Ctile，所需的shared mem size又减小一半。
